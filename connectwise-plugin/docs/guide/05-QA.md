# Questions & answers

The questions real users actually ask, with straight answers.

## "The client says they replied, but I don't see it"

Give it one sync cycle (usually under a minute). Still nothing? Check the
**Dashboard** — is the client's connection badge green and is *Last Sync*
advancing? If the badge is red, the credentials or the network to ConnectWise
are the problem, and the log line will say which.

## "I logged 30 minutes but ConnectWise shows something else"

It shouldn't — 30 minutes arrives as exactly 0.50 h. If you see an old
ticket where hours look shifted or doubled, it predates the current version;
new entries are exact to four decimals. (And yes, "2 hours" as `120`
minutes is fine.)

## "Why does ConnectWise credit the time to Basit when Dalbeir did the work?"

By design. The client sees their **assigned technician** — you don't expose
your internal team. Open the ⏱ popup on the osTicket side to see who
*really* worked: the Technician column and the orange/blue source chips tell
the true story for your own reporting.

## "The dropdown is missing a status / has one I don't want"

The Status and Time Type dropdowns mirror the client's ConnectWise lists —
if their admin adds or renames one, press ↻ Refresh on the client's edit
page and osTicket follows. *Open* and *Closed* are the only extras; osTicket
itself requires them and they still behave correctly (Closed completes the
ConnectWise ticket).

## "Do ticket assignments sync between the systems?"

**No — by design.** ConnectWise is the system of record: clients create and
assign tickets there using their own queues and resources. osTicket imports
by the configured filters (queues, companies, resources) — *not* by matching
technicians — and any of your technicians can work any imported ticket.
Their updates flow back to the ConnectWise ticket, credited to its assigned
resource. There is no technician mapping to set up or maintain.

## "A file didn't arrive"

Check, in order: **Sync attachments** is on for that client → the file is
under ConnectWise's ~6 MB API limit (bigger files get a link-note in the thread
instead, so nothing is lost) → the Dashboard log for an upload error on that
ticket.

## "Someone edited a time entry in ConnectWise — what happens here?"

The ⏱ popup and totals update to the new duration/billable flag. The
original thread entry stays as history — billing records are never silently
rewritten. Deleted in ConnectWise? Same: popup reflects it, history stays.

## "Could a sync hiccup create duplicates?"

No. Every synced ticket stores both its ConnectWise ID and its osTicket ID, so
updates always target the existing ticket — and every reply, note, time
entry and file carries its own fingerprint ID. Run a sync a hundred times;
you get exactly one of each. This is tested to paranoia levels (the test
suite syncs everything three times and counts).

## "How do I verify the integration is working?"

The five-minute confidence check, end to end:

1. Create a ticket in ConnectWise → it appears in osTicket with the customer,
   priority, status and any files.
2. Reply from osTicket → the reply shows on the ConnectWise ticket.
3. Log **30 minutes billable** and **15 minutes non-billable** → ConnectWise
   shows 0.50 h billable and 0.25 h non-billable, and the ⏱ popup total
   matches ConnectWise's Time Summary.
4. Attach a file on each side → it's downloadable on the other.
5. Change the status in ConnectWise → the osTicket status follows by name
   (and completing the ticket closes it here).

For a deeper check, run `selftest.php` — 27 automated verdicts including a
live API round-trip.

## "The blue panel / a new feature isn't showing"

Hard-refresh first (Ctrl+F5). If it's still old, the `scp/` endpoint copies
are stale — re-copy them (Setup, Step 1) and refresh again.

## "Something is genuinely wrong — where do I look?"

1. **Dashboard → Recent Logs**, filter *Errors*. The messages are written
   for humans and name the ticket involved.
2. **Failed & Dead Jobs** on the same page — one click retries them after
   you fix the cause.
3. Still stuck? Run the health check and read its verdicts:
   `php include/plugins/<plugin-dir>/selftest.php`
4. For deep dives, set Log Level to *debug* temporarily — payloads are
   recorded with secrets automatically blanked out.
